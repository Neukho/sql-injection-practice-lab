SQL Injection을 눈으로 보고 손으로 익히는 실습 사이트

실제 데이터베이스 없이 브라우저 안에서만 동작하는 교육용 랩이다. 사용자가 입력한 문자열이 쿼리 의미를 바꾸면 왜 인증 우회와 정보 노출이 생기는지, 그리고 같은 입력도 parameterized query에서는 왜 무력화되는지를 한 화면에서 비교한다.

학습 흐름: 입력이 들어오면 쿼리 구조가 바뀌고, 그 결과 인증 또는 조회 결과가 달라진다. 방어는 이 연결을 끊는 데서 시작한다.

Unsafe Query

SELECT * FROM users
WHERE username = 'admin' --'
AND password = 'anything';

Safe Rewrite

SELECT * FROM users
WHERE username = ?
AND password = ?;
문자열 결합 vs Prepared Statement

What You Learn

실습에서 바로 보는 세 가지 핵심

  • 입력값이 SQL 구문으로 섞이는 순간 왜 조건 검사가 무너지는지 읽기 쉽게 추적한다.
  • 같은 문자열도 safe query에서는 단순 데이터로만 취급된다는 점을 즉시 비교한다.
  • 결과 패널에서 원인, 쿼리 변화, 결과, 방어 포인트를 순서대로 읽는다.

UI/UX Support

학습이 막히지 않도록 화면 구조부터 설계

이 페이지는 예쁜 데모보다 학습 효율을 우선한다. 그래서 비교가 필요한 정보는 같은 화면에 붙여 놓고, 다음에 무엇을 봐야 하는지도 바로 보이게 설계했다.

Comparison First

취약 쿼리와 안전 쿼리를 나란히 배치

학습자는 같은 입력이 두 모드에서 어떻게 다르게 해석되는지 즉시 비교할 수 있다.

Guided Reading

결과는 원인부터 읽게 구성

실습 결과는 원인, 변화, 결과, 방어 순서로 정리되어 논리 흐름이 끊기지 않는다.

Accessible Flow

짧은 문장과 명확한 상태 표시

모드 상태, 샘플 payload, 결과 제목을 크게 보여 줘서 처음 보는 사람도 현재 위치를 잃지 않는다.

Learning Flow

원인에서 방어까지 한 줄로 이어지는 학습 구조

SQL injection은 어려운 기술이 아니라 입력이 쿼리 구조를 바꾸는 문제다. 그래서 이 사이트는 개념 소개보다 먼저, 원인과 결과가 어떻게 연결되는지 논리적으로 보여준다.

01 Cause

원인

개발자가 입력값을 문자열로 이어 붙이면 사용자는 데이터 대신 쿼리 조각을 넣을 수 있다.

02 Query Shift

쿼리 변화

따옴표, 주석, UNION, 세미콜론이 들어가면 WHERE 조건과 반환 구조가 바뀐다.

03 Result

결과

인증 우회, 전체 목록 노출, 다른 테이블 정보 노출, 데이터 손상 위험으로 이어진다.

04 Defense

방어

parameterized query를 쓰면 입력은 구문이 아니라 값으로 처리되어 인과 연결이 끊긴다.

Applied Constraints

강화된 하네스 프롬프트 제약을 구현에 반영

이 사이트는 프롬프트 단계에서 정한 안전 제약을 그대로 구현에 옮겼다. 실제 실행이 아니라 규칙 기반 시뮬레이션만 허용하고, 모든 데이터는 브라우저 내부 mock data로만 처리한다.

Runtime Boundary

실제 서버와 DB 없음

fetch, API, DB, local storage 없이 현재 페이지 메모리 상태만 사용한다.

Execution Boundary

실행이 아닌 패턴 시뮬레이션

payload는 정규식과 mock 시나리오 규칙으로만 판정되며 실제 SQL은 실행되지 않는다.

Input Boundary

입력은 정규화 후 표시

입력값은 길이 제한, 개행 제거, escape 처리 후 화면과 시뮬레이션 상태에 반영된다.

Interactive Lab

실습 콘솔

모드를 바꾸고 샘플 payload를 눌러보면 취약한 쿼리와 안전한 쿼리가 동시에 바뀐다. 결과는 모두 mock data를 기준으로 안전하게 시뮬레이션되며, 설명은 원인과 결과가 바로 연결되도록 작성했다.

    Attack Patterns
    
            
    실습에서 확인할 SQLi 패턴
    
          
    
          
    
            사이트는 실제 공격을 수행하지 않는다. 대신 어떤 문자열이 왜 위험하고, 어떤 결과를 유발하는지 명확하게 설명한다.
          
    
        
    

        
    
          
    
            
    01
    
            
    
              
    Tautology
    
              
    항상 참이 되는 조건을 넣어 WHERE 절 전체를 무력화하는 패턴이다.
    
            
    
            
    예: 인증 우회, 전체 목록 노출
    
          
    

          
    
            
    02
    
            
    
              
    Comment Termination
    
              
    뒤에 오는 조건을 주석 처리해 비밀번호 검증 같은 절을 끊어버리는 방식이다.
    
            
    
            
    예: admin' --
    
          
    

          
    
            
    03
    
            
    
              
    UNION SELECT
    
              
    원래 조회 결과에 다른 테이블의 데이터를 합쳐 정보 노출을 유도하는 패턴이다.
    
            
    
            
    예: 계정 목록, 이메일, 역할 정보 노출
    
          
    

          
    
            
    04
    
            
    
              
    Stacked Query
    
              
    세미콜론 이후에 추가 명령을 이어 붙여 데이터 변경이나 삭제 위험을 만드는 패턴이다.
    
            
    
            
    실습에서는 실행하지 않고 위험성만 시뮬레이션
    
          
    
        
    
      
    
    
    

    
    
      
    
        
    
          
    
            
    Developer Lens
    
            
    개발자는 어떤 코드를 먼저 봐야 하는가
    
          
    
          
    
            SQL injection은 보통 복잡한 해킹 기술이 아니라, 서버 코드에서 입력을 문자열로 이어 붙인 결과다.
            그래서 개발자에게 필요한 것은 공격 기법 암기보다 취약 코드 위치를 빠르게 찾는 감각이다.
          
    
        
    

        
    
          
    
            
    Code Review
    
            
    먼저 찾을 것
    
            
      
              
    • 문자열 결합으로 SQL을 만드는 코드
      • 
              
    • 검색, 로그인, 관리자 조회처럼 입력이 바로 붙는 기능
      • 
              
    • DB 오류를 그대로 노출하는 처리
      • 
            
    
          
    
          
    
            
    Refactor
    
            
    바로 바꿀 것
    
            
      
              
    • Prepared statement 또는 ORM parameter binding
      • 
              
    • 최소 권한 DB 계정 분리
      • 
              
    • 로그와 모니터링 기준 추가
      • 
            
    
          
    
          
    
            
    Mindset
    
            
    기억할 것
    
            
    입력값을 신뢰하지 않는 것이 아니라, 입력값이 쿼리 구조를 바꿀 수 없게 만드는 것이 핵심이다.
    
          
    
        
    
      
    
    
    

    
    
      
    
        
    
          
    
            
    Learner Guide
    
            
    SQL injection을 공부하는 사람이 꼭 챙겨야 할 질문
    
          
    
          
    
            처음 공부할 때는 payload를 외우기보다, 입력이 쿼리에 섞였을 때 어떤 조건이 바뀌는지를 읽는 연습이 더 중요하다.
          
    
        
    

        
    
          
    
            
    Question 01
    
            
    입력이 어디에 들어갔는가
    
            
    로그인인지 검색인지에 따라 깨지는 조건이 다르다. 먼저 입력 위치를 확인한다.
    
          
    
          
    
            
    Question 02
    
            
    쿼리 구조가 어떻게 바뀌었는가
    
            
    주석, OR 조건, UNION, 세미콜론 중 무엇이 WHERE 절이나 반환 구조를 바꿨는지 본다.
    
          
    
          
    
            
    Question 03
    
            
    왜 safe mode에서는 실패하는가
    
            
    같은 문자열도 구문이 아니라 값으로 전달되면 공격이 아니라 단순 입력으로 남는다.
    
          
    
        
    
      
    
    
    

    
    
      
    
        
    
          
    Defense Guide
    
          
    방어는 "문자열을 코드로 해석하지 않게 만드는 것"에서 시작한다
    
          
    
            SQL injection 방어의 핵심은 입력값을 구문이 아니라 데이터로 처리하는 것이다.
            parameterized query는 같은 입력이 들어와도 SQL 구조를 바꾸지 못하게 막기 때문에,
            문자열 결합보다 훨씬 설득력 있는 1차 방어선이 된다. 여기에 최소 권한, 오류 메시지 통제, 로깅을 함께 적용해야 실제 운영 환경에서 버틸 수 있다.
          
    
          
      
            
    • 문자열 결합으로 SQL을 만들지 않는다.
      • 
            
    • Prepared statement와 ORM parameter binding을 우선 사용한다.
      • 
            
    • DB 계정 권한을 최소화해 stacked query 피해 범위를 줄인다.
      • 
            
    • DB 오류를 사용자에게 직접 노출하지 않는다.
      • 
            
    • 로그인, 검색, 관리자 기능에 대한 이상 징후를 로깅하고 모니터링한다.
      • 
          
    
        
    

        
    
          
    
            
    취약한 예시
    
            
    const sql =
  "SELECT * FROM users WHERE username = '" +
  username +
  "' AND password = '" +
  password +
  "'";
    
          
    
          
    
            
    안전한 예시
    
            
    const sql =
  "SELECT * FROM users WHERE username = ? AND password = ?";

db.execute(sql, [username, password]);
