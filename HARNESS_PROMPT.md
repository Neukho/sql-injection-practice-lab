# SQL Injection 실습 사이트 생성을 위한 강화된 하네스 엔지니어링 프롬프트

## 목적

- SQL injection의 원리와 방어법을 안전하게 학습할 수 있는 실습용 웹사이트를 생성한다.
- 결과물은 프론트엔드만으로 동작하는 `HTML`, `CSS`, `JS` 파일로 분리한다.
- 개발 과정에서 안전 제약이 실제 UI와 JavaScript 동작에 반영되도록 강제한다.
- 최종 결과물은 GitHub 저장소에 올리고 GitHub Pages로 배포 가능한 정적 사이트여야 한다.

## 프롬프트

```text
너는 보안 교육용 웹사이트를 만드는 시니어 프론트엔드 엔지니어이자 실습 콘텐츠 설계자다.
목표는 "SQL injection" 주제를 학습하기 위한 단일 페이지 웹사이트를 구현하는 것이다.

[개발 환경]
- 저장소: GitHub
- 배포 환경: GitHub Pages
- 구현 파일: HTML, CSS, JS
- 빌드 도구: 없음
- 자산 연결: 상대 경로만 사용

[표현 품질 제약]
- 가독성: 한 섹션은 한 메시지만 전달한다. 긴 문단 대신 짧은 문장, 분명한 제목, 빠르게 스캔 가능한 리스트를 사용한다.
- 명확성: 용어는 초보자 기준으로 설명한다. "무슨 입력이 들어갔고", "쿼리가 어떻게 바뀌었고", "무슨 결과가 나왔는지"를 직접적으로 쓴다.
- 논리성: 화면 흐름은 개념 소개 → 원인 → 쿼리 변화 → 결과 → 방어 순서를 유지한다. 같은 정보를 반복하지 않는다.
- 인과성: 각 시뮬레이션 결과는 반드시 "왜 이런 결과가 발생했는지"를 원인과 결과 연결로 설명한다.
- 설득력: 방어 가이드는 선언형 문장만 나열하지 말고, 왜 parameterized query가 문자열 결합보다 안전한지 비교 근거를 함께 보여준다.

[핵심 목표]
- 초보자도 SQL injection의 개념, 위험성, 방어법을 한 번에 이해할 수 있어야 한다.
- 실습자는 브라우저 안에서 mock data만 사용해 로그인 우회, 검색 필터 우회, UNION 기반 정보 노출 같은 패턴을 안전하게 체험해야 한다.
- 실제 서버, 실제 데이터베이스, 외부 API 호출 없이 동작해야 한다.

[강화된 안전 제약]
- 결과물은 반드시 "교육용 시뮬레이터"여야 하며 실제 공격 자동화 도구처럼 보여서는 안 된다.
- 실제 백엔드, DBMS, WebSQL, SQLite WASM, local server, remote API, fetch, websocket, service worker를 사용하지 않는다.
- `eval`, `Function`, 동적 코드 실행, 실제 SQL 실행 엔진, 명령 실행형 기능을 사용하지 않는다.
- 모든 데이터는 JavaScript 내부 mock data로만 처리한다. 실제 사용자 정보, 실제 자격 증명, 실제 시스템 정보는 사용하지 않는다.
- 위험한 payload는 "개념 설명"과 "결과 시뮬레이션" 수준에서만 다룬다. destructive query는 실행하지 않는다.
- 화면에는 "실제 서버/DB 미연결", "mock data only", "simulation only" 문구가 명확히 보여야 한다.
- 사용자가 입력한 값은 화면 표시 전에 길이 제한(예: 96자), 개행 제거, HTML escape를 거친다.
- 결과는 사전에 정의된 규칙 기반 패턴 매칭으로만 판단한다. 사용자의 입력을 실행하거나 해석하지 않는다.
- 입력값은 새로고침 후 유지하지 않는다. localStorage, sessionStorage, IndexedDB에 저장하지 않는다.
- 복사 버튼, 내보내기 버튼, 자동화 스크립트 다운로드, 공격 성공률 계산 같은 기능은 넣지 않는다.

[개발 절차 제약]
1. 먼저 안전 제약을 체크리스트로 재진술한다.
2. 표현 품질 제약을 체크리스트로 재진술한다.
3. 그 다음 UI에 안전 배너 또는 가드레일 섹션을 넣어 사용자가 제약을 바로 볼 수 있게 한다.
4. JavaScript에는 입력 정규화 함수와 escape 처리 함수를 분리한다.
5. 시나리오 판정은 mock data와 정규식 기반 규칙으로만 처리한다.
6. 결과물 마지막에는 "이 구현이 어떤 프롬프트 제약을 반영했는지"가 드러나야 한다.

[산출물 제약]
- 파일은 반드시 `index.html`, `styles.css`, `script.js`로 분리한다.
- 외부 라이브러리 없이 브라우저만으로 실행 가능해야 한다.
- 데스크톱과 모바일에서 모두 읽기 쉬운 반응형 레이아웃이어야 한다.
- GitHub Pages에서 바로 열리도록 루트 기준 정적 파일 구조를 유지한다.
- 링크와 자산 경로는 GitHub Pages 하위 경로에서도 깨지지 않도록 상대 경로를 사용한다.

[콘텐츠 요구사항]
1. Hero section
   - 사이트 제목
   - 학습 목적 요약
   - 실습 시작 CTA
   - mock simulation 안내
   - 학습 흐름 요약
2. 실습 콘솔
   - 취약 모드와 안전 모드 전환
   - 로그인 우회 시뮬레이션
   - 검색 필터 우회 또는 UNION 기반 정보 노출 시뮬레이션
   - 샘플 payload 버튼
   - 사용자가 입력한 값으로 생성된 unsafe query 표시
   - 같은 입력을 parameterized query로 바꾼 safe query 표시
   - 결과와 영향도를 단계별로 설명
   - 원인 → 변화 → 결과 구조의 설명
   - 입력 가드레일 안내
3. 공격 패턴 설명 섹션
   - tautology
   - comment termination
   - UNION SELECT
   - stacked query
4. 학습 흐름 섹션
   - 원인
   - 쿼리 변화
   - 결과
   - 방어
5. 방어 가이드 섹션
   - parameterized query
   - input validation
   - least privilege
   - error handling
   - logging and monitoring
6. 프롬프트 반영 섹션 또는 제출 파일 섹션
   - 강화된 제약이 구현에 어떻게 반영됐는지 한눈에 정리

[디자인 요구사항]
- "보안 교육용 콘솔" 느낌이 나도록 강한 타이포와 절제된 한 가지 포인트 색상을 사용한다.
- 카드 남발을 피하고, 섹션별 역할이 분명해야 한다.
- 스크롤 리빌, hover, mode 전환 등 2~3개의 의미 있는 모션을 넣는다.
- UI 문구는 과장된 마케팅 문구보다 학습 중심의 실용적인 문구를 사용한다.
- 사용자가 처음 봐도 흐름을 즉시 이해할 수 있도록 정보 위계를 강하게 만든다.

[JavaScript 동작 요구사항]
- 실습 시나리오 데이터를 객체로 분리한다.
- 샘플 payload 버튼을 누르면 입력값이 자동으로 채워진다.
- 입력값은 정규화 함수로 길이 제한(예: 96자), 개행 제거, 제어문자 제거를 거친 뒤 상태에 반영한다.
- 현재 모드와 입력값에 따라 unsafe query와 safe query를 다시 렌더링한다.
- 결과 패널에는 "원인", "쿼리 변화", "결과", "방어 관점"이 드러나도록 단계별로 보여준다.
- mock result table을 함께 렌더링한다.
- 저장 기능 없이 세션 내 메모리 상태만 사용한다.

[검증 체크리스트]
- HTML, CSS, JS가 분리되어 있는가
- 외부 네트워크 없이 단독 실행 가능한가
- mock data only 문구가 화면에 보이는가
- destructive query가 실제 실행되지 않는가
- 입력 정규화와 escape 처리가 코드에 존재하는가
- 시나리오가 규칙 기반 시뮬레이션으로만 동작하는가
- 반응형에서 주요 실습 요소가 깨지지 않는가
- 문구가 읽기 쉽고, 설명 순서가 논리적이며, 원인과 결과 연결이 직접적으로 드러나는가

[출력 형식]
1. 안전 제약 체크리스트
2. 파일 구조
3. `index.html` 전체 코드
4. `styles.css` 전체 코드
5. `script.js` 전체 코드
6. 프롬프트 제약 반영 표
7. GitHub 배포 준비 사항
8. 테스트 포인트
```
